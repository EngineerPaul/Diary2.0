Redis 1)
При запуске контейнера redis вылетает предупреждение "WARNING Memory overcommit must be enabled!...".
Эта ошибка требует установить настройку vm.overcommit_memory = 1 в линуксе снаружи контейнера.
Это необходимо из-за механизма фонового сохранения redis, который сохраняет копии, что увеличивает объем памяти.
Если при работе не будет возникать нехватки памяти, то это не приведет к проблемам.
https://serverfault.com/questions/1053313/is-it-possible-to-set-vm-overcommit-memory-1-only-for-specific-containers
https://stackoverflow.com/questions/76830848/redis-warning-memory-overcommit-must-be-enabled

Redis 2)
При создании нового пользователя в redis начинает вылетать ошибка "no permissions to access a channel redis".
Для ее решения добавил какие-то права пользователю: &* в docker-compose.yml
Основная настройка - https://habr.com/ru/articles/823936/
Доп. права - https://github.com/celery/celery/issues/8724

Celery 1)
Celery worker и beat должны иметь доступ к файлам проекта, а проект соответственно к celery_client.
Вместе с этим все requirements должны присутствовать во всех этих контейнерах.
По этой причине для всех трех контейнеров (tgserver, worker, beat) создан один dockerfile.

Celery 2)
При обычном запуске celery в контейнере вылатет предупреждение, что использовать root пользователя недопустимо.
Поэтому создаем нового пользователя (сразу для трех контенеров из-за общего dockerfile)

Celery 3)
Из-за создания нового пользователя celery beat контейнер начинвает выдавать ошибку о нехватке прав пользователя.
Это вызвано тем, что при запуске beat создается файл celerybeat-schedule, в котором сохраяется дата последнего бита.
Новый же пользователя не имеет прав на создание файлов. Создание файла вручную ни к чему не привели.
Для решения добавил в dockerfile создание папки beatdata и передачу прав собственности нового пользователю.
Дополнительно в celery_client необходимо указать, что файл теперь лежит в папке (beat_schedule_filename).
